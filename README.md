# 일학습병행 정보 미러링 대시보드

공동훈련센터가 담당 기업들의 일학습병행 서류(사업계획서/전담인력등록) 제출 현황을 확인·승인하고, 기업은 서류를 첨부해 공단(PDMS) 제출을 준비하는 정보 미러링 시스템입니다.

## 주요 기능

- **정보 가시성**: 기업별 과정 진행 현황, 승인 상태, 주요 이슈를 실시간으로 확인
- **필수서류 3단계 검증**: 파일명 매칭 → 문서 내용 확인 → 기준 서식 대조로 "확인됨/누락/불일치" 판정 (AI가 자동 승인하지 않음, 최종 판단은 사람)
- **승인/반려 워크플로우**: 센터가 서류를 검토해 승인하거나, 반려 사유를 댓글로 남겨 기업에 보완 요청
- **댓글 피드백 & 이메일 알림**: 서류 추가/수정 요청을 댓글로 전달, 새 댓글은 이메일로도 알림
- **PDF 서류 첨부 및 빈 양식 다운로드**: 사업계획서·전담인력등록 PDF 첨부/조회, 빈 양식(.docx)은 별도 다운로드
- **이슈 자동 감지**: 전담인력 누락/중복 이슈 자동 계산
- **초대·코드 기반 가입**: 셀프 회원가입 없음 — 기업은 센터 초대, 센터는 가입 코드로만 계정 생성
- **센터 다중화**: 센터마다 소속 회사만 조회/관리 (다른 센터 데이터는 접근 불가)

## 기술 스택

- **Frontend**: Next.js 16 (App Router, Turbopack), TypeScript, Tailwind CSS
- **Backend**: Supabase (PostgreSQL, Auth, Realtime, Storage)
- **Authentication**: 이메일/비밀번호 (초대·코드 기반 가입, 셀프 회원가입 없음)
- **이메일 발송**: Resend
- **PDF 생성**: pdfkit (데모 서류) / **빈 양식 생성**: docx (자료실 양식)

## 실행 방법

### 1. 의존성 설치
```bash
npm install
```

### 2. 환경 변수 설정
`.env.local` 파일에 아래 값을 설정하세요:
```
NEXT_PUBLIC_SUPABASE_URL=your_supabase_url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_supabase_anon_key

# 센터 셀프 가입 시 입력해야 하는 비밀 코드 (서버에서만 비교, 브라우저에 노출 안 됨)
CENTER_INVITE_CODE=아무도-못-맞출-랜덤-문자열

# 기업 담당자 초대 메일 발송(auth.admin.inviteUserByEmail), 센터 가입 코드 확정 등
# 관리자 작업에 필요 - Supabase 대시보드 Project Settings > API > service_role
SUPABASE_SERVICE_ROLE_KEY=your_service_role_key

# 댓글/승인·반려 알림 메일 발송 - resend.com 가입 후 API Keys에서 발급
# 없으면 이메일 발송만 조용히 스킵되고 나머지 기능은 정상 동작
RESEND_API_KEY=your_resend_api_key
```

`SUPABASE_SERVICE_ROLE_KEY`는 RLS를 완전히 우회하는 관리자 권한 키입니다. 절대 클라이언트에 노출되면 안 되며 (`NEXT_PUBLIC_` 접두사 금지), server-only 코드(`src/lib/supabase-admin.ts`)에서만 사용합니다.

### 3. 데이터베이스 설정
Supabase SQL Editor에서 아래 순서대로 실행하세요:

1. `supabase-schema.sql` — 기본 테이블(companies, courses, profiles, comments)과 RLS 정책 생성
2. `supabase-rls-policies.sql` — 비로그인(anon) 사용자의 조회를 차단 (인증된 사용자만 읽기 가능하도록 강화)
3. `supabase-remove-default-center-role.sql` — 가입 트리거 수정: 신규 가입자는 role 없음(권한 없음)으로 시작, 초대/코드 승인 시에만 role 부여
4. `supabase-multi-center.sql` — `centers` 테이블 신설 및 센터 다중화(회사가 특정 센터에 소속되도록 격리)

모두 재실행해도 안전하게 작성되어 있습니다.

### 4. 개발 서버 실행
```bash
npm run dev
```
브라우저에서 http://localhost:3000 으로 접속하세요.

## 가입 & 권한 체계

셀프 회원가입은 없습니다 (예전엔 이메일/비밀번호만 입력하면 가입되고 자동으로 센터 권한이 부여되는 심각한 보안 문제가 있었습니다 — 아래 "보안 조치 이력" 참고). 지금은 역할별로 가입 경로가 다릅니다.

- **센터 담당자**: `/login`의 "센터 가입"에서 이메일/비밀번호와 함께 **센터명**, **센터 가입 코드**(`CENTER_INVITE_CODE`)를 입력합니다. 코드는 서버(`/api/claim-center-role`)에서만 비교되고 브라우저에는 절대 노출되지 않습니다. 센터명은 오타로 중복 생성되는 걸 줄이기 위해 제출 전 재확인 단계를 거치며, 이미 같은 이름의 센터가 있으면 합류하고 없으면 새로 만듭니다(find-or-create).
- **기업 담당자**: 셀프 가입 불가. 센터가 `/companies` 화면의 "기업 담당자 초대" 패널에서 이메일과 회사명을 입력하면 Supabase 초대 메일이 발송됩니다(`/api/invite-user`, `auth.admin.inviteUserByEmail`). 회사명도 직접 입력이며, 내 센터 소속에 없는 이름이면 새 회사로 자동 생성됩니다. 초대 수락 시 role/company_id가 메타데이터로 실려 있어 가입 즉시 자동 반영되고, `/auth/set-password`에서 비밀번호를 설정하면 완료됩니다.
- **최초 센터 계정 부트스트랩**: 위 흐름은 "초대할 센터가 이미 있어야" 성립하므로, 시스템에 센터가 하나도 없는 최초 상태는 Supabase 대시보드에서 수동으로 만들어야 합니다 (Authentication → Users → Invite user, 이후 Table Editor에서 `profiles.role`을 `center`로 설정).

## 필수서류 3단계 검증

기존엔 파일명 키워드 매칭만으로 "제출 완료"를 판정했습니다 (문서 내용은 전혀 보지 않음). 지금은 3단계로 검증합니다.

1. **1단계 — 파일명 매칭** (`src/lib/document-validation.ts`, 순수 함수, 클라이언트/서버 공용): 기업명·과정명 키워드로 후보 파일을 찾습니다. **확정 판단이 아닙니다.**
2. **2단계 — 문서 내용 확인** (`src/lib/document-validation-server.ts`, `server-only`): PDF 텍스트를 추출해 문서 종류(사업계획서 vs 전담인력등록)가 맞는지 확인합니다.
3. **3단계 — 기준 서식 대조**: `src/lib/document-schemas.ts`에 정의된 필수 항목들이 실제로 문서에 있는지 대조합니다.

결과는 항상 **확인됨 / 누락 / 불일치** 3단계로만 리포트되며, `/api/validate-document`를 통해 클라이언트에 노출됩니다. **AI가 자동으로 승인하지 않습니다** — 대조 결과만 제시하고 최종 서류 적합성 판단은 담당자가 직접 합니다. 이 원칙은 `.claude/skills/document-validation/SKILL.md`에 문서화되어 있습니다.

## 서류 자료실 & 데모 서류

- **데모 서류 PDF**: 실제 "일학습병행 학습기업 신청 공고"(PDMS) 서식을 참고해 `scripts/generate-demo-pdfs.mjs`로 생성합니다. 서식 탭 헤더, 표 테두리, 대표자명/사업자등록번호, 제출일자·서명란(신청기업명/대표 성명(인)/한국산업인력공단 귀중)을 포함합니다. 같은 과정 묶음(사업계획서+전담인력등록)은 날짜를 통일했습니다.
- **빈 양식(자료실)**: 기업이 다운로드해서 작성할 빈 양식은 `scripts/generate-templates-docx.mjs`로 **`.docx`**(워드/한글에서 편집 가능) 형식으로 생성합니다 — PDF는 편집이 어려워 선택했습니다. 대시보드 상단 "자료실" 섹션에서 다운로드할 수 있습니다.
- **제출된 서류 다운로드**: 자료실의 빈 양식과는 별개로, 이미 첨부된 서류 옆의 "다운로드" 버튼은 **실제로 채워진 파일**을 받아 공단 제출 사이트(PDMS)에 바로 첨부할 수 있게 합니다.

## 승인/반려 워크플로우

`CourseApprovalActions.tsx`로 센터 전용 승인/반려 버튼을 제공합니다. 승인 시 과정 상태가 실제로 변경되고 자동 댓글이 남으며, 반려 시 입력한 사유가 댓글로 기업에 전달됩니다. 기업 포털은 실시간 구독으로 상태 변경을 즉시 반영하고, 승인되면 "PDMS 바로가기" 배너가 노출됩니다.

## 댓글 & 이메일 알림

- **첨부 요청 버튼**: 전담인력 이슈 시 센터가 기업에 첨부 요청
- **담당자 확인 버튼**: 전담인력 중복 시 확인 요청

댓글이 하나 저장(`INSERT`)될 때마다 클라이언트가 `/api/notify-comment`를 fire-and-forget으로 호출하고,
서버가 그 댓글의 과정(course) → 기업(company) → **소속 센터**를 조회해서 "같은 회사 담당자 전원 +
그 센터 담당자 전원"에게 이메일을 보냅니다. **작성자 본인은 항상 제외**됩니다. 발송은 Resend(`src/lib/email.ts`)를
쓰고, `RESEND_API_KEY`가 없으면 에러 없이 조용히 스킵됩니다(이메일 실패가 댓글 저장을 막지 않음).
댓글 **수정/삭제는 알림이 가지 않습니다** (INSERT 시점에만 발송).

이메일이 발송되는 지점 7곳:

| 화면 (컴포넌트) | 트리거 |
|---|---|
| 센터 (`CommentSection`) | 일반 댓글 작성 |
| 센터 | "첨부 요청" 버튼 |
| 센터 | "담당자 확인" 버튼 |
| 기업 (`CompanyCommentSection`) | 일반 댓글 작성 |
| 기업 | "조치 완료" 버튼 |
| 센터 (`CourseApprovalActions`) | 승인 버튼 (자동 댓글) |
| 센터 | 반려 확정 (자동 댓글, 반려 사유 포함) |

## 이슈 자동 감지

- **전담인력 누락**: 전담인력등록 서류가 없으면 자동 감지
- **전담인력 중복**: 동일 기업 내 여러 과정이 같은 전담인력 파일을 사용하면 자동 감지

## 보안 조치 이력

실사용 테스트 중 발견되어 수정한 문제들입니다.

| 문제 | 내용 | 조치 |
|---|---|---|
| 자동 센터 권한 부여 | 회원가입만 하면 자동으로 `role: center` 부여 → 전체 기업 데이터 열람 가능 | 가입 트리거 수정, 신규 가입자는 role 없음(권한 없음)으로 시작 |
| `/setup` 페이지 | 로그인 없이 누구나 "A/B/C사 담당자" 역할을 스스로 선택해 부여받을 수 있던 사칭 경로 | 페이지 삭제 |
| 로그아웃 후 데이터 잔존 | `signOut()`이 `router.refresh()`만 호출하고 페이지 이동을 안 시켜서, 로그아웃해도 이전 화면 데이터가 그대로 남음 | `/login`으로 강제 이동하도록 수정 |
| 센터 페이지 role 미검증 | `/companies`가 로그인 여부만 확인하고 role은 안 봐서, 기업 계정도 URL만 알면 전체 기업 데이터 열람 가능 | `role === 'center'` 검증 추가 |
| 셀프 회원가입 전체 | 이메일/비밀번호만으로 누구나 가입 가능 (Google OAuth 포함) | 셀프 가입·Google OAuth 제거, 초대/코드 기반으로 전환 |
| 센터 데이터 미격리 | 센터가 하나뿐이라고 가정한 설계라, 신규 센터가 기존 센터와 동일한 전체 회사 데이터를 봄 | `centers` 테이블 신설, 센터별 소속 회사로 조회/초대 범위 격리 |
