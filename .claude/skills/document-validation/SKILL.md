# 필수서류 검증 원칙

이 프로젝트(HRD-Bridge)에서 "필수서류가 제출되었는가"를 판별하는 코드를 작성하거나 수정할 때는 아래 3단계 검증을 순서대로 거쳐야 한다. 파일명만 보고 확정 판단을 내리는 코드는 이 스킬이 다루는 영역에서는 통과시키지 않는다.

## 1단계 — 파일명/메타데이터 매칭 (1차 분류용, 확정 판단 금지)

- 파일명, 업로드 시 지정된 `file_type` 라벨, 회사명·과정명 키워드 매칭 등은 **후보를 추리는 1차 필터**로만 사용한다.
- "파일명이 매칭됐다 = 서류가 정상 제출됐다"로 결론 내리지 않는다. 1단계 결과는 항상 2단계로 넘어가는 입력일 뿐, 최종 판정이 아니다.
- 매칭 실패(파일 자체가 없음)는 이 단계에서 바로 "누락"으로 보고해도 되지만, 매칭 성공은 아직 "확인됨"이 아니다.

## 2단계 — 문서 내용(텍스트) 확인

- 1단계를 통과한 파일은 실제 텍스트를 추출(PDF 파싱 등)해서, 필수 섹션/항목이 본문에 실존하는지 확인한다.
- 예: "전담인력등록" 파일이라면 이름, 소속, 담당 업무 등 필수 항목 텍스트가 실제로 문서 안에 있는지 확인한다. 파일명만 그럴듯하고 내용이 빈 문서이거나 엉뚱한 서류인 경우를 잡아내는 단계.

## 3단계 — 기준 서식(스키마) 대조

- 추출한 내용을 사전에 정의된 기준 서식/스키마와 대조하여 필수 항목의 누락 여부를 항목 단위로 점검한다.
- 스키마는 서류 종류별로 명시적으로 정의되어야 하며(예: 전담인력등록 스키마, 사업계획서 스키마), 하드코딩된 파일 경로 매핑으로 대체하지 않는다.

## 결과 리포트 형식

검증 결과는 항상 아래 3단계 중 하나로만 보고한다. 이 3개 외의 상태(예: "위험", "경고", 색상만으로 구분되는 상태 등)를 새로 만들지 않는다.

| 상태 | 의미 |
|---|---|
| 확인됨 | 1~3단계를 모두 통과, 필수 항목이 실제 문서 내용으로 확인됨 |
| 누락 | 서류 자체가 없거나(1단계 실패), 문서는 있으나 필수 섹션/항목이 내용에 없음(2·3단계 실패) |
| 불일치 | 서류는 있고 관련 내용도 있으나, 기준 서식과 값이 다르거나 항목이 어긋남(3단계에서 매칭은 되지만 스키마 위반) |

## 금지사항 — AI 자동 승인/자동 판단 금지

- 이 검증 로직은 **대조 결과(확인됨/누락/불일치)를 제시하는 것까지만** 담당한다.
- "자동으로 승인 처리", "자동으로 이슈를 닫음", "AI가 최종 합격/불합격을 결정" 하는 로직은 만들지 않는다.
- 최종 판단(서류가 요건을 충족하는지, 승인할지)은 항상 사람이 내린다. 시스템·AI는 판단을 대신하지 않고 판단에 필요한 대조 결과만 제공한다.
- 코드 리뷰나 구현 중 이 경계를 넘는 로직(예: 검증 결과에 따라 상태를 자동으로 "승인"으로 바꾸는 코드)을 발견하면 바로 지적한다.

## 현재 코드베이스와의 차이 (알아둘 것)

현재 `getPdfFiles`([src/app/companies/[id]/page.tsx](../../../src/app/companies/%5Bid%5D/page.tsx))와 `my-company/page.tsx`의 동일 로직은 파일명/키워드 매칭(1단계)만으로 "전담인력 등록 누락" 여부를 확정하고 있으며, 2·3단계 검증은 아직 없다. 이 영역을 개선할 때는 위 3단계 구조로 리팩터링하는 것을 우선 고려한다.
