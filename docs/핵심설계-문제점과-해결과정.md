# HRD-Bridge 핵심 설계 — 문제점과 해결과정

HRD(인적자원개발) 지원사업 참여 기업의 필수서류(사업계획서, 전담인력등록 등) 제출 현황과 이슈를
**공동훈련센터 ↔ 참여기업** 간에 실시간으로 공유하는 정보 미러링 대시보드.

- **스택**: Next.js 16 (App Router) · React 19 · TypeScript · Tailwind CSS 4 · Supabase (Auth / PostgreSQL + RLS / Realtime) · Resend(이메일) · pdf-parse(문서 검증)
- **데이터 모델**: `centers` → `companies` → `courses` → `course_files` / `comments`, 사용자 역할은 `profiles.role`(`center` | `company`)
- **역할 구도**: 센터는 소속 기업 현황을 조회·피드백(댓글)·승인/반려하고, 기업은 자기 회사 과정의 서류를 제출·보완한다. 최종 판단은 항상 사람이 한다.

---

## 1. 아키텍처 한눈에 보기

```
[기업 담당자]                        [센터 담당자]
    │  파일 업로드 / 댓글                │  조회 · 댓글 · 승인/반려
    ▼                                   ▼
┌─────────────────── Next.js (App Router) ───────────────────┐
│  클라이언트 컴포넌트         서버(API Route / RSC)           │
│  - 실시간 구독(Realtime)     - /api/validate-document        │
│  - 댓글/파일 UI              - /api/invite-user (초대 발송)  │
│  - 검증 결과 표시            - /api/claim-center-role        │
│                              - /api/notify-comment (메일)    │
└──────────────┬───────────────────────────┬─────────────────┘
               ▼                           ▼
        Supabase (Auth + Postgres RLS + Realtime)      Resend(메일)
```

핵심 설계 축은 4가지다.

1. **3단계 문서 검증 파이프라인** — 파일명 매칭 → 문서 내용 확인 → 기준 서식(스키마) 대조
2. **역할 기반 폐쇄형 가입 체계** — 기업은 초대 전용, 센터는 가입 코드 검증
3. **실시간 미러링** — Supabase Realtime(`postgres_changes`)으로 파일/댓글 변경을 양쪽 화면에 즉시 반영
4. **센터 다중화 + RLS 격리** — 센터는 자기 소속 기업만, 기업은 자기 회사만 조회 가능

## 2. 문제점과 해결과정

### 문제 1. "파일명만 보고 서류 제출 완료로 판단"하던 검증의 한계

**문제점**
초기 구현은 파일명·키워드 매칭만으로 "전담인력 등록 누락" 여부를 확정했다.
그러나 실제 운영을 가정하면 파일명만 그럴듯한 빈 문서, 엉뚱한 서류, 다른 과정의 서류가
첨부되어도 "정상 제출"로 통과되는 구조였다. 반대로 시스템이 자동으로 합격/불합격을
판정하게 두면 오판 책임 문제도 생긴다.

**해결과정**
검증을 **3단계 파이프라인**으로 재설계하고, 각 단계의 역할과 한계를 명시했다.

| 단계 | 내용 | 위치 |
|---|---|---|
| 1단계 | 파일명/키워드 매칭 — **후보 분류만**, 확정 판단 금지 (매칭 실패만 즉시 '누락') | `src/lib/document-validation.ts` |
| 2단계 | PDF 텍스트 추출 → 문서 제목 키워드 + **실제 기업명/과정명 대조** | `src/lib/document-validation-server.ts` |
| 3단계 | 서류 종류별 스키마(`DOCUMENT_SCHEMAS`)와 대조해 필수 항목 누락 점검 | `src/lib/document-schemas.ts` |

결과는 **확인됨 / 누락 / 불일치** 3가지 상태로만 리포트하고, "AI/시스템의 자동 승인·자동
판단은 만들지 않는다"를 프로젝트 원칙으로 문서화했다(`.claude/skills/document-validation/SKILL.md`).
시스템은 대조 결과만 제시하고 최종 판단은 항상 사람이 한다.

**배운 점** — 검증 로직은 "얼마나 잘 잡아내는가"만큼 "어디까지만 판단하는가"의 경계 설계가 중요하다.

### 문제 2. 다른 회사·다른 과정의 서류를 잘못 첨부해도 통과되는 문제 + 한국어 띄어쓰기 오탐

**문제점**
2단계에서 문서 제목("사업계획서", "전담인력")만 확인하면 **다른 기업/다른 과정의
서류를 첨부해도 제목이 같아 통과**된다. 그래서 문서 본문에 실제 기업명·과정명이
등장하는지 대조를 추가했는데, 이번에는 한국어 띄어쓰기 표기 차이("데이터분석" vs
"데이터 분석") 때문에 정상 서류가 '불일치'로 오탐되는 문제가 생겼다.

**해결과정**
- 기업명/과정명 대조를 2단계(문서 내용 확인)에 편입시켜, 제목이 맞아도 대상(기업/과정)이 다르면 '불일치'로 분류.
- 대조 전 **공백을 아예 제거하고 비교**(`stripSpaces`)해서 띄어쓰기 표기 차이로 인한 false '불일치'를 제거했다. 일반 키워드 매칭은 공백을 정규화(`normalize`)하는 것과 구분해 적용했다.

```ts
// 한국어는 "데이터분석"/"데이터 분석"처럼 띄어쓰기가 표기에 따라 달라지므로
// 공백 유무를 다른 내용으로 취급하면 안 된다.
const stripSpaces = (text: string) => text.replace(/\s+/g, '')
```

**배운 점** — 텍스트 대조 검증은 언어 특성(한국어 띄어쓰기 유연성)을 고려하지 않으면 정탐보다 오탐이 먼저 터진다.

### 문제 3. PDF 파싱 라이브러리의 서버/클라이언트 경계 문제

**문제점**
검증 로직을 한 파일에 모아두니 클라이언트 컴포넌트가 PDF 파싱 코드(`pdf-parse`,
`node:fs`)까지 import하게 되어 번들 에러가 났다. 게다가 `pdf-parse`의 `index.js`에는
CJS "메인 모듈" 판별 로직이 있어 **동적 import 시 자체 테스트 픽스처를 읽으려다
오작동**하는 라이브러리 자체 버그도 있었다.

**해결과정**
- 모듈을 **순수 함수(클라이언트 공용) / 서버 전용**으로 물리적으로 분리:
  - `document-validation.ts` — 1단계 순수 함수만. Node API 사용 금지 (클라이언트에서도 import 가능)
  - `document-validation-server.ts` — 2·3단계. 첫 줄에 `import 'server-only'`를 선언해 **실수로 클라이언트 번들에 포함되면 빌드 타임에 즉시 에러**가 나게 안전장치를 걸었다.
- 브라우저는 `/api/validate-document` 엔드포인트를 통해서만 2·3단계 결과를 받는다.
- `pdf-parse`는 `index.js` 대신 실제 구현 파일을 정적 import해 CJS 판별 버그를 우회했다.

```ts
import 'server-only'
import pdfParse from 'pdf-parse/lib/pdf-parse.js' // index.js의 CJS 메인 모듈 판별 버그 우회
```

**배운 점** — Next.js에서 서버 전용 코드는 "주의해서 안 쓰기"가 아니라 `server-only` 패키지로 빌드 타임에 강제해야 안전하다.

### 문제 4. 초대 메일 링크가 클릭도 하기 전에 만료되는 `otp_expired` 미스터리

**문제점**
기업 담당자 초대 메일의 가입 링크를 사용자가 클릭하면 `otp_expired` 에러가 떴다.
토큰 만료 시간을 늘려도 소용없었다. 원인은 **Gmail 등의 보안 스캐너가 메일 수신 직후
링크를 미리 방문(프리페치)해 1회용 토큰을 먼저 소모**해버리는 것 — 사용자가 클릭할 땐
이미 사용된 토큰이었다.

**해결과정**
링크 방문(GET)과 토큰 검증을 분리했다.

1. 메일에는 Supabase `action_link`를 직접 넣지 않고, 토큰 해시만 쿼리로 실은 자체 페이지 `/auth/accept-invite?token_hash=...`로 보낸다.
2. 이 페이지는 **정적으로 로드될 뿐 토큰을 소모하지 않는다.** 스캐너가 방문해도 아무 일도 일어나지 않는다.
3. 사용자가 "초대 수락" **버튼을 직접 클릭해야만** `supabase.auth.verifyOtp()`가 호출되어 토큰이 검증된다. (스캐너는 버튼을 클릭하지 않는다)

**배운 점** — 이메일의 1회용 링크는 "GET 방문 = 사용자 의도"가 아니다. 부수효과 있는 동작은 반드시 명시적 사용자 액션 뒤로 옮겨야 한다.

### 문제 5. Supabase 기본 초대 메일의 한계 (영문 템플릿 · localhost 노출)

**문제점**
`inviteUserByEmail`의 기본 메일은 영문 템플릿에 프로젝트 Site URL 설정을 그대로 쓰기
때문에, 센터명/회사명을 본문에 넣을 수 없고 로컬 개발 중 저장된 `localhost` 주소가
실제 초대 메일에 노출되는 문제가 있었다.

**해결과정**
발송 파이프라인을 둘로 쪼갰다.
- **계정 생성 + 토큰 발급**: `admin.auth.admin.generateLink({ type: 'invite', ... })` 까지만 Supabase에 맡기고,
- **실제 발송**: 인증된 자체 도메인(`attude.uk`)의 Resend 커스텀 템플릿으로 직접 보낸다. 센터명·회사명을 본문에 넣고, 링크는 요청 origin 기준으로 생성해 localhost 노출을 없앴다.
- 초대 시 역할·소속(`role: 'company'`, `company_id`)을 **사용자 메타데이터에 실어 보내고**, 가입 완료 시 `handle_new_user()` DB 트리거가 이를 `profiles`에 자동 반영한다.

**배운 점** — BaaS의 기본 메일 기능은 프로토타입용이다. 실서비스 품질이 필요하면 "토큰 발급"과 "발송"을 분리해 발송만 직접 가져오면 된다.

### 문제 6. 역할별로 가입 경로가 완전히 달라야 하는 폐쇄형 가입 체계

**문제점**
아무나 가입해 센터/기업 데이터를 보면 안 된다. 그런데 역할마다 신뢰 모델이 다르다 —
기업 담당자는 "센터가 아는 사람"이고, 센터 담당자는 "코드를 아는 내부자"다.
또 가입 확정 시점에 RLS 정책 상태에 따라 DB 쓰기가 실패하는 문제도 겪었다.

**해결과정**
가입 경로를 역할별로 이원화했다.

| 역할 | 가입 경로 | 신뢰 근거 |
|---|---|---|
| 기업 | 셀프 가입 불가. 센터가 `/api/invite-user`로 초대해야만 계정 생성 | 센터의 명시적 초대 |
| 센터 | 셀프 가입 + `/api/claim-center-role`에서 **가입 코드 검증** | `CENTER_INVITE_CODE` (서버 환경변수, 브라우저로 전송되지 않음) |

- 코드 비교는 서버에서만 수행하고 정답값은 절대 클라이언트로 내려보내지 않는다.
- 신규 가입자의 기본 role은 NULL(권한 없음)로 두어, 초대 메타데이터나 코드 검증 없이는 아무것도 못 보게 했다.
- 가입 확정 단계의 DB 쓰기(센터/회사 find-or-create, 프로필 갱신)는 **관리자(service role) 클라이언트**로 수행한다. 코드 검증을 통과해야만 도달하는 이미 신뢰된 경로이므로, RLS 정책이 어떤 상태든 항상 동일하게 동작해야 하기 때문이다.
- 센터명/회사명은 자유 입력 + find-or-create로 처리하되, 오타로 인한 중복 생성은 재입력 확인 단계로 줄였다(완벽히 막지는 못함을 인지하고 문서화).

**배운 점** — "가입"은 하나의 기능이 아니라 역할별 신뢰 모델의 집합이다. 신뢰가 확인된 서버 경로에서는 RLS에 의존하지 말고 관리자 권한으로 결정적으로 처리하는 편이 안정적이다.

### 문제 7. 센터가 여러 개가 되면서 생긴 데이터 격리 문제 (센터 다중화)

**문제점**
초기 설계는 센터가 1개라는 암묵적 가정 위에 있었다. `companies`/`profiles`에 소속 센터
정보가 아예 없어서, 센터가 늘어나면 **모든 센터가 모든 기업을 볼 수 있는** 구조였다.

**해결과정**
- `centers` 테이블을 도입하고 `companies.center_id`, `profiles.center_id`를 추가. 기존 데이터는 마이그레이션 SQL로 단일 센터에 이관했다.
- 조회 권한을 앱 코드가 아닌 **RLS 정책으로 강제**했다: 센터는 `profiles.center_id = companies.center_id`인 기업만, 기업은 `profiles.company_id = companies.id`인 자기 회사만 SELECT 가능. `courses`도 소속 회사를 경유해 같은 기준으로 제한.
- 마이그레이션 SQL은 **재실행해도 안전하게**(`IF NOT EXISTS`, `DROP POLICY IF EXISTS` 후 재생성) 작성해 정책 변경 이력을 파일로 관리했다(`supabase-multi-center.sql` 등).

**배운 점** — 권한 격리는 UI에서 숨기는 게 아니라 DB 계층(RLS)에서 걸어야 하고, "지금은 1개"라는 가정은 스키마에 명시적 소속 컬럼이 없으면 나중에 반드시 비용으로 돌아온다.

### 문제 8. 실시간 미러링 — Realtime 이벤트 누락과 낙관적 업데이트 중복

**문제점**
기업이 파일을 올리면 센터 화면에, 센터가 댓글을 달면 기업 화면에 즉시 보여야 한다.
Supabase Realtime의 `postgres_changes`를 서버측 filter를 걸어 구독하면 이벤트가
안정적으로 수신되지 않는 경우가 있었고, 본인이 작성한 댓글은 (낙관적 UI 추가 +
Realtime INSERT 수신)으로 **중복 표시**되는 문제가 있었다.

**해결과정**
- 구독은 **filter 없이 테이블 전체를 구독하고, 코드에서 `course_id`로 필터링**하는 방식으로 전환해 수신 안정성을 확보했다.
- 낙관적으로 추가한 댓글이 Realtime으로 다시 들어와도 중복되지 않도록, **id 기준 존재 확인 후 추가**하는 방어 로직을 넣었다.

```ts
setComments((prev) =>
  prev.some((c) => c.id === newCommentData.id) ? prev : [...prev, newCommentData])
```

- 파일 첨부/삭제도 `course_files` 테이블 구독으로 INSERT/UPDATE/DELETE를 각각 반영하고, 서류 상태 변경 시 검증(2·3단계)을 자동 재실행해 "주요 이슈" 표시가 실시간으로 갱신되게 했다.

**배운 점** — 실시간 UI는 "이벤트 수신"보다 "이벤트가 두 번 오거나 안 올 때"의 방어 설계(멱등한 상태 갱신)가 본체다.

### 문제 9. 이메일 알림이 핵심 기능을 방해하지 않게 하는 설계

**문제점**
댓글 작성 시 상대방(같은 회사 담당자 + 소속 센터 담당자)에게 이메일 알림을 보내는데,
메일 발송 실패(API 키 미설정, Resend 장애 등)가 댓글 저장 자체를 실패시키면 안 된다.
또 알림 API를 아무나 호출해 남의 과정 댓글 내용을 메일로 뿌리게 두어서도 안 된다.

**해결과정**
- 알림은 **fire-and-forget**: 클라이언트가 댓글 insert 성공 후 별도로 호출하고, 발송 실패 시에도 200으로 응답한다. `sendEmail`은 예외를 던지지 않고 `{ sent, reason }` 결과만 반환한다.
- 발송 전 **호출자가 해당 과정에 실제 접근 권한이 있는지**(본인 회사 또는 본인 센터) 서버에서 검증하고, 작성자 본인은 수신자에서 제외한다.
- 발신 주소는 Resend에 인증된 자체 도메인으로 고정해 임의 수신자에게도 실제 발송되게 했다.

**배운 점** — 부가 기능(알림)은 핵심 기능(데이터 저장)과 실패 도메인을 분리해야 한다. 단, "부가"라고 권한 검증까지 생략하면 정보 유출 통로가 된다.

## 3. 설계 원칙 요약 (한 줄씩)

1. **파일명 매칭은 분류이지 판정이 아니다** — 판정은 내용(2단계)과 스키마(3단계)가 한다.
2. **시스템은 대조 결과(확인됨/누락/불일치)까지만** — 승인/반려는 항상 사람이 결정한다.
3. **서버 전용 코드는 `server-only`로 빌드 타임에 강제**한다.
4. **1회용 토큰은 명시적 클릭 뒤에서만 소모**한다 (메일 스캐너 프리페치 방어).
5. **권한 격리는 UI가 아니라 RLS**로, 신뢰된 서버 경로는 관리자 클라이언트로.
6. **실시간 상태 갱신은 멱등하게** — 이벤트 중복/누락을 전제로 설계한다.
7. **부가 기능의 실패가 핵심 기능을 막지 않게** 실패 도메인을 분리한다.
